What is the law surrounding this topic?

European Union directive 2016/2102 says that the website and mobile applications for public sector bodies need to conform to WCAG 2.0 Level AA.

New websites must comply from 23 September 2019.

Old websites must comply from 23 September 2020.

Moblie applications must comply from 23 June 2021.

WCAG 2.0 was published in 2008. WCAG 2.1 was published in 2018.

WCAG 2.1 doesn't replace WCAG 2.0, it just adds to it.

In January 2012 the Royal National Institute of the Blind (RNIB) started legal action against the low-cost airline Bmibaby because their website couldn't be used.

Time-based Media

üìΩ üîä

Basically, provide alternatives for video or audio.

Only giving cliff-notes for this since we don't do a lot of it.

For Level A compliance, this just means text alternatives. Song lyrics, script for the training video, paragraphs below the video containing the video's text etc.

Level AA covers captions for live video and audio description for videos.

Whereas A allows text alternative or audio description (1.2.3) AA needs audio description (1.2.5).

Level AAA covers things like sign language, extended audio descriptions and audio-only live video alternatives.

Extended audio description can pause the video to describe a scene.

We can work around a different layout but it makes things harder for us.

We (frontenders) need to worry about focus and tab order.

We shouldn't need both vertical and horizontal scrollbars to see all the content.

Unless it's essential to convey the meaning, we shouldn't need a second scrollbar for:

  • Vertical scrolling content at a width of 320 pixels
  • Horizontal scrolling content at a height of 256 pixels

We should be able to zoom in to 200% without loss of functionality.

Trello is an example of horizontal scrolling content.

320px = 1280px at 400%; 256px = 1024px at 400%

Consider form fields - "first" & "last" on same line, "last" hidden when zoomed!

Responsive layouts are usually zoomable by design.

Text

üìù

Rules about copy, vocabulary and text layout.

We should be able to apply these styles without any loss of functionality.


.any-text {
    letter-spacing: 0.12em; /* Tracking */
    line-height:    1.5;    /* Line spacing */
    margin-bottom:  2em;    /* Spacing after paragraphs */
    word-spacing:   0.16em; /* Word spacing */
}

All values are relative to the font-size of the text.

It should be possible to programmatically determine the purpose of the link from the link text alone.

Level AAA Techniques

Try to avoid using jargon or idioms. Provide a way for explaining them like a glossary or definition list.

Add a mechanism for expanding abbreviations or include them in the text.

Generally the page is too complex for someone with a lower secondary education level then provide a summary or graphs/diagrams.

Roughly 11 years old / Year 7.

WCAG 1.4.8 (AAA) For a large block of text, the user should be able to do the following:

  1. Change the foreground and/or background colours.
  2. Limit the width to 80 characters (40 if CJK).
  3. Prevent the text being justified.
  4. Expand line spacing to at least space-and-a-half within paragraphs and spacing after paragraphs to at least 1.5 times larger than the line spacing.
  5. Expand the text to 200% without assistive technology and without losing functionality.

CJK = Chinese, Japanese or Korean.

Justified = left & right aligned.

UX and UI

üíØ

How the user should work with the website.

The navigation and identification should be consistent throughout the site.

This doesn't mean that the navigation can't expand/collapse, just that the order and position don't change between pages.

Identification means things like similar terms or icons.

There should be multiple ways of working out where you are within a set of pages.

Ways we can achieve this are:

  • Sitemap
  • Site search
  • Table of contents
  • Links ("Previous" and "Next" etc.)

Location can be found through breadcrumb trail or a highlighted link (e.g. in the sidebar).

We don't need all of these ways, one will do.

Location is Level AAA.

There shouldn't be a "change of context" when an element gains focus or an input is used.

Basically, if you want to go somewhere or do something, use a button.

Remember the early 2000's when using a dropdown to change page was cool?

If you need to do something through gesture, give us a button to do it as well.

For example: stretch-to-zoom on Google Maps can also be done with the plus button.

Any custom UI element should have a target size of at least 44 × 44 pixels unless:

  • There's a equivalent that's large enough.
  • It's inline in a block of text.
  • It's User Agent controlled (checkboxes etc.).
  • The size is essential to the information being conveyed.

We will assume that you intented native checkboxes, radio buttons, dropdowns and scrollbars.

Exception for dropdowns: Select2

This is more of a request than an actual WCAG order.

Please stop adding carousels to designs.

For a carousel, we need to add functionality like making it pausable or letting the user adjust the time.

We need to prevent links in hidden slides being clickable and focusable.

We need to announce when content changes.

They're also distracting.

They're just really hard to get right.
