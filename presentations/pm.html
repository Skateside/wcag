What the law says

What is the law surrounding this topic?

European Union directive 2016/2102 says that the website and mobile applications for public sector bodies need to conform to WCAG 2.0 Level AA.

New websites must comply from 23 September 2019.

Old websites must comply from 23 September 2020.

Moblie applications must comply from 23 June 2021.

Public sector does not cover industrial and commercial (us).

WCAG 2.0 was published in 2008. WCAG 2.1 was published in 2018.

WCAG 2.1 doesn't replace WCAG 2.0, it just adds to it.

In January 2012 the Royal National Institute of the Blind (RNIB) started legal action against the low-cost airline Bmibaby because their website couldn't be used.

In January 2019, Beyonc√©'s Parkwood Entertainment was sued over website accessibility because it voilated the Americans With Disabilities Act (1990).

So there is legal precedent for this sort of thing.

When the page has been build, we need to be able to apply these styles to any text without a loss of functionality.

Designers might start spacing out the text a little more.

"Loss of functionality" means not being able to click a button, for example.

We should try to avoid using jargon or idioms. Provide a way for explaining them like a glossary or definition list.

If the page is too complex for someone with a lower secondary education level then there should be a summary or graphs/diagrams.

If the pronounciation of a word is ambiguous then we should include an explanation.

Widgets

üé∞

Users need to be able to interact with the things we build the way they want to.

The biggest change is that widgets need to be keyboard accessible.

For example, anything that would appear on hover should also appear on focus.

Keyboard navigation extends to the entire site. We also need to be able to see what we're focussed on.

We might need longer to develop/QA a widget.

outline:none; can be added if it's replaced by something!

We won't add these styles if the site's style sheet removed them.

There are some techniques we can use to hide the outline from mouse but preserve for keyboard.

Any pointer gesture needs to have a complementary button.

A good example is stretch-to-zoom on Google Maps can be done with the zoom-in button.

The same goes for motion gestures (the iPhone "shake to undo" feature can be replaced with the delete button on the virtual keyboard).

Think of pointer/motion gestures as optional extras.

To get AAA compliance, buttons need to be at least 44 × 44 pixels unless:

  • There's a equivalent that's large enough.
  • It's inline in a block of text.
  • It's User Agent controlled (checkboxes etc.).
  • The size is essential to the information being conveyed.

Custom input inside a <label> might count as "inline".

You might see buttons getting bigger.

Because of the "User Agent controlled" exception, we will assume that designers intended these elements to be native:

  • Checkboxes
  • Radio Buttons
  • Dropdowns *
  • Scrollbars

* Unless the dropdown is clearly meant to be Select2.

There are a lot of tests to do to make sure custom UI elements are up-to-scratch.

If these are meant to be custom, you need to let us know.

For AAA compliance, a change of context needs to be user initiated.

For example, filtering shouldn't be applied until the user presses a "search" button.

Change of context = significant change to the page, such as search results or page navigation.

As a result, widgets might have buttons when they didn't before.

Carousels need to die.

There are so many rules we need to apply to get them right.

This isn't a WCAG requirement, just a developer's plee.

Please try to talk clients out of having a carousel. We'll make a carousel work if we have to.

Miscellany

üç≤

Other points that I couldn't put into a group of their own.

There need to be text alternatives to images and videos.

For images, that's as simple as adding an alt attribute.

Videos will need transcripts if they contain any speech or text.

There are alt equivalents for icons and SVG's.

Videos should have the option of captions if they have speech.

There are rules on background audio.

If a sound plays for more than 3 seconds, the user should be allowed to pause/stop it.

We don't get many requests for this, but just in-case someone asks.

For AAA compliance, if there is background audio that:

  1. is primarily foreground speech,
  2. is not audio for CAPTCHA and
  3. is not vocalisation as expression

then is at least one of these should be true:

  • There are no background sounds
  • The background sounds can be turned off.
  • Background sounds are at least 20 dB lower than the foreground.

This is Level AAA.

Vocalised expression = singing, rapping etc.

20 dB exceptions: occasional sounds lasting only 1 or 2 seconds.

We shouldn't need both vertical and horizontal scrollbars to see all the content.

Unless it's essential to convey the meaning, we shouldn't need a second scrollbar for:

  • Vertical scrolling content at a width of 320 pixels
  • Horizontal scrolling content at a height of 256 pixels

We should be able to zoom in to 200% without loss of functionality.

Trello is an example of horizontal scrolling content.

320px = 1280px at 400%; 256px = 1024px at 400%

Consider form fields - "first" & "last" on same line, "last" hidden when zoomed!

Responsive layouts are usually zoomable by design.
