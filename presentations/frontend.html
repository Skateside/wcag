What the law says

üëÆ‚Äç‚ôÄÔ∏è üëÆ‚Äç‚ôÇÔ∏è

What is the law surrounding this topic?

European Union directive 2016/2102 says that the website and mobile applications for public sector bodies need to conform to WCAG 2.0 Level AA.

New websites must comply from 23 September 2019.

Old websites must comply from 23 September 2020.

Moblie applications must comply from 23 June 2021.

Public sector does not cover industrial and commercial (us).

WCAG 2.0 was published in 2008. WCAG 2.1 was published in 2018.

WCAG 2.1 doesn't replace WCAG 2.0, it just adds to it.

In January 2012 the Royal National Institute of the Blind (RNIB) started legal action against the low-cost airline Bmibaby because their website couldn't be used.

In January 2019, Beyonc√©'s Parkwood Entertainment was sued over website accessibility because it voilated the Americans With Disabilities Act (1990).

So there is legal precedent for this sort of thing.

Building webpages

üï∑ üï∏

Here's what the guidelines say about making web pages.

Pages should have a <title> tag.

Pro tip: put the company name at the end of the title.


<!-- Instead of this: -->
<title>Xigen Design - About us</title>

<!-- Do this: -->
<title>About us - Xigen Design</title>

This prevents all the icons looking the same when we have a lot of open tabs.

The "pro tip" isn't part of WCAG, just a good idea.

Use WAI-ARIA landmark roles to identify content areas.


<!-- Header, Logo etc. -->
<header role="banner"></header>
<!-- Primary navigation -->
<nav role="navigation"></nav>
<!-- Success/error alerts -->
<div role="alert"></div>
<!-- Hero - no role (assuming no <h1> in the hero) -->
<section></section>
<!-- Main content -->
<main role="main"></main>
<!-- Footer, copyright etc. -->
<footer role="contentinfo"></footer>

This is Level A

The hero assumes that the <h1> is not in there.

Only applies to overall page.

We should be able to bypass content that's repeated across pages.

WAI-ARIA landmark roles are a way to do that. We could also use "Skip to content" links or collapsed elements.

Make sure collapsed content is actually hidden - more in Widgets.


<body>
    <ul class="nav-access" aria-label="Short-cuts">
        <li>
            <a href="#content" class="nav-access__link">
                Skip to main content
            </a>
        </li>
    </ul>
    <!-- header, nav, hero -->
    <main id="content" role="main" aria-label="Main content">
        <!-- ... -->
    </main>
    <!-- teasers, footer -->
</body>

This is a Level AAA technique.

aria-current has a few possible values.

It's the design team's job to include breadcrumbs in the design - it's our job to make them work.


/* Was: */
.sidebar a.is-active {
    /* ... */
}

/* Now: */
.sidebar a[aria-current="page"] {
    /* ... */
}

These selectors have the same selector strength and efficiency. They work in everything we need them to.

Attribute selectors started working in IE7.

For HTML:

  • Elements that need them should have complete start and end tags.
  • Nest according to specs (only within valid parents, only contain valid children).
  • No duplicate attributes.
  • IDs are unique.

tl;dr write valid HTML.

Consider validating.

Some aria-* attributes are only valid inside a certain role.

This extends to semantic meaning of elements - always use the right element for the job.

But beware of CSS removing the semantic meaning.


/* This removes the semantic meaning of tables */
table {
    display: block;
}

Sadly, putting display:table; isn't enough to put it back.

Anchors go somewhere, buttons do something.

Styling the tables to stack on mobile is a good way of adding display:block.

Most other display values have the same effect.

To preserve the meaning, you need role attributes.

                <!-- ... -->
            </tr>
        </thead>
        <tbody role="rowgroup">
            <tr role="row">
                <td role="cell">Ipsum</td>

We shouldn't need both vertical and horizontal scrollbars to see all the content.

Unless it's essential to convey the meaning, we shouldn't need a second scrollbar for:

  • Vertical scrolling content at a width of 320 pixels
  • Horizontal scrolling content at a height of 256 pixels

We should be able to zoom in to 200% without loss of functionality.

Trello is an example of horizontal scrolling content.

320px = 1280px at 400%; 256px = 1024px at 400%

Consider form fields - "first" & "last" on same line, "last" hidden when zoomed!

Responsive layouts are usually zoomable by design.

Pages need a language

HTML 5


<!doctype html>
<html lang="fr">

XHTML 1.1


<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.1//EN"
   "http://www.w3.org/TR/xhtml11/DTD/xhtml11.dtd">
<html xmlns="http://www.w3.org/1999/xhtml" xml:lang="fr">

For XHTML transitional, use lang and xml:lang.

The language should be a lower-case ISO2 code of the language.


<html lang="en">
<html lang="fr">
<html lang="da">
<html lang="el">

It can include a script subtag to define the script that's being used.


<!-- Japanese written in Katakana -->
<html lang="ja-Kana">
<!-- Russian written in Cyrillic -->
<html lang="ru-Cyrl">

It can include a region subtag. Usually that's the upper-case ISO2 country code.


<html lang="en-GB">
<html lang="fr-CA">
<html lang="da-DK">
<html lang="el-GR">

... but not always


<!-- Central American Spanish -->
<html lang="es-013">

All the parts can be combined but they're always in the order language-script-region.


<!-- Russian written in Cyrillic spoken in Belarus -->
<html lang="ru-Cyrl-BY">

The language alone is usually enough.

If you're unsure about the ISO2 code then look it up.

Add lang to any different language except for:

  • Technical terms. ("Nasopharyngitis")
  • Proper names. ("Vladimir Putin")
  • Indeterminate language. ("Lorem ipsum dolor ...")
  • Words/phrases that are part of the vernacular. ("status quo")

If you need to move things around, consider using BreakPoints.


var breakpoints = new BreakPoints({
    md: 576
    // ...
});

var nodeToMove = document.querySelector("...");
var parentNode = nodeToMove.closest("...");

breakpoints.on("change", "md", function (e) {

    if (e.detail.matches) {
        parentNode.appendChild(nodeToMove);
    } else {
        parentNode.insertBefore(nodeToMove, parentNode.firstChild);
    }

}, true);

Don't just re-order things with CSS because the focus/tab order will be wrong.

https://gist.github.com/james-jlo-long/fe69c667463d127063617a2b4d5a54c5

You may be thinking that you can set the tab order using tabindex.

Don't do it - you'd have to do it for every focusable element!

There are 2 exceptions:

tabindex="-1"
I can gain focus but I'm not in the tab order.
tabindex="0"
I am part of the natural tab order.

We should be able to tell which element has focus when we tab through the page.

Don't add this unless there's something to replace it.


a {
    outline: none;
}

Anchors and buttons should have a hover and focus state.


a:hover,
a:focus {
    /* ... */
}

If you need to remove the outline for mouse, keep it for keyboard!


/* twitter.com/leaverou/status/1045768279753666562 */
:focus:not(:focus-visible) {
    outline: none;
}

Inputs

‚úçÔ∏è

Form fields could use some care and attention.

Make sure inputs have labels.


<label for="input-id">Descriptive label</label>
<input type="text" id="input-id">

<input type="text" aria-label="Descriptive label">

<p id="text-id">Descriptive text</p>
<input type="text" aria-labelledby="text-id">

Just putting the label in as the placeholder text isn't enough - use one of the techniques above.

Adding a sr-only class to the <label> is fine.

You don't need id/for if you nest your input.


<label>
    <input type="checkbox" name="tandc">
    I have read the terms and conditions
</label>

But it's always better to include it if you can.


<label for="tandc">
    <input type="checkbox" name="tandc" id="tandc">
    I have read the terms and conditions
</label>

WAI: Generally, explicit labels are better supported by assistive technology.

Don't worry too much about styling some inputs. It's far easier to leave these as native elements:

  • Checkboxes
  • Radio buttons
  • Dropdowns *
  • Scrollbars

* Except for Select2.

Admittedly, scrollbars don't count as inputs.

Widgets

üé∞

Users should be able to interact with the things we build the way they want to.

First rule of accessible widgets: if there's a native element that does the job - use it.

<a> goes somewhere, <button> does something.


<!-- Don't do this -->
<h3 role="button">Section heading</h3>

<!-- This wouldn't make sense -->
<h3 role="heading button" aria-level="3">Section heading</h3>

<!-- Do this instead -->
<h3>
    <button type="button">Section heading</button>
</h3>

You might see this in footers the become collapsible.

We should be able to operate our widgets with the keyboard if we want.

There should be no keyboard trap.

If your widget has custom keyboard short-cuts then one of the following is true:

  • You can turn it off.
  • You can remap them to one or more non-printable characters.
  • Short-cuts are only active when widget is focussed.

Not sure how this would affect things like the Christmas game.

Pointer gestures (things like "stretch to zoom") should be mapped to single pointer gestures (things like "click to zoom in").

This is the same for motion gestures ("shake to undo").

The user should be able to cancel a single pointer gesture.

This means listen for things like click or keypress/input rather than mousedown or keydown.

Click only counts if mousedown and mouseup are on the same element.

Content that appears on hover should also appear on focus. (1.4.13 - AA)

  • The user should be able to dismiss the content without moving the mouse.
  • The user should be able to hover over the content without the content disappearing.
  • The content should remain visible until the hover/focus is lost, the user dismisses it or the information is no longer valid.

Bootstrap has Tooltips and Popovers - both fail these tests.

Make sure that hidden stuff is actually hidden.


.not-hidden {
    height: 0;
    overflow: hidden;
}

.also-not-hidden {
    visibility: hidden;
    /* opacity: 0; */
}

.actually-hidden {
    display: none;
}
        

<div class="not-hidden" aria-hidden="true" inert>
    Really hidden.
</div>

Not hidden means screen readers still see the content and links are still in the tab order.

Option: listen for transitionend and toggle a class to set display; none;.

inert requires a polyfill.

Tappable things should be fat-finger friendly. This means a button should be at least 44 × 44 pixels unless:

  • There's an equivalent that's big enough.
  • It's inline with a block of text.
  • The size is determined by the User Agent.
  • Being that small is essential to convey the meaning.

44 × 44 is a AAA requirement.

"Determined by the User Agent" = native inputs.

Not sure if <label> around a checkbox counts, but probably does.

There shouldn't be a "change of context" when an element gains focus or an input is used.

Basically, if you want to go somewhere or do something, use a button.

Remember the early 2000's when using a dropdown to change page was cool?

They should work with mouse, finger and keyboard.

The user should be able to pause the auto-play, change the time to up to 10 times the default, or hide the carousel on any device.

It should have the appropriate role and aria-* attributes so screen readers can understand it.

Distinguishable

üêà üêï

Users should be able to tell the difference between foreground and background.

Usually this is the design team's job but sometimes we need to add alerts.

Don't just use colour to show information.

Don't just do this:


<input type="text" class="is-error">

This is a little better.


<input type="text" aria-invalid="true">

This is the best.


<label for="input-id">Descriptive label</label>
<input type="text" id="input-id" aria-invalid="true" aria-errormessage="error-id">
<label for="input-id" id="error-id">Descriptive error message</label>

Screen readers don't see classes but they will see aria-invalid.

jQuery Validate does "best" (without aria-errormessage) by default.

Alerts are another good colour gotcha.


<div class="alert alert-error">Lorem ipsum dolor ...</div>

Make sure there's something visually distinct. Mark it as an alert for screen readers as well.

Animations

üéû

Animations aren't always as accessible as you'd think.

Consider whether animation is necessary.

Some users prefer not to see animations. We can detect that in CSS.


@media screen and (prefers-reduced-motion) {
    .my-element {
        transition: none;
    }
}

As of March 2019 prefers-reduced-motion only works in Firefox and Safari.

Some iOS 7 users reported motion sickness and vertigo from some of the animations.

Don't make anything flash more than 3 times a second.

A simple transition less than 0.3 seconds isn't a problem since it changes and stays.
