What the law says

What is the law surrounding this topic?

European Union directive 2016/2102 says that the website and mobile applications for public sector bodies need to conform to WCAG 2.0 Level AA.

New websites must comply from 23 September 2019.

Old websites must comply from 23 September 2020.

Moblie applications must comply from 23 June 2021.

Public sector does not cover industrial and commercial (us).

WCAG 2.0 was published in 2008. WCAG 2.1 was published in 2018.

WCAG 2.1 doesn't replace WCAG 2.0, it just adds to it.

In January 2012 the Royal National Institute of the Blind (RNIB) started legal action against the low-cost airline Bmibaby because their website couldn't be used.

Page Building

üèó

Some basic rules for building pages.

Web pages need to be valid HTML.

Pages need a <title> tag.

Pro tip: put the company name at the end of the title.


<!-- Instead of this: -->
<title>Xigen Design - About us</title>

<!-- Do this: -->
<title>About us - Xigen Design</title>

The "pro tip" isn't part of WCAG. It stops all the tabs looking the same when they're many.

Might be important to know if you're integrating a template.

Form errors should be connected to the input.

Prevailing wisdom is to make them a <label> element.


<label for="input-id">Descriptive label</label>

<input type="text" id="input-id"
    aria-invalid="true"
    aria-errormessage="error-id">

<label for="input-id" id="error-id">
    Descriptive error message
</label>

Because of the for/aria-errormessage association, the label can be anywhere on the page.

Page navigation should always be in the same place and the same order.

However, sub-navigation can be added/removed.


<nav>
    <ul>
        <li><a href="...">Home</a></li>
        <li>
            <a href="..." aria-current="page">What is this?</a>
            <ul>
                <li><a href="...">Why am I here?</a></li>
                <li><a href="...">How did you get here?</a></li>
                <li><a href="...">Can I go now?</a></li>
            </ul>
        </li>
        <li><a href="...">Contact us</a></li>
    </ul>
</nav>

It can change between users, but the order & location should be consistent.

Pages need a language

HTML 5


<!doctype html>
<html lang="fr">

XHTML 1.1


<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.1//EN"
   "http://www.w3.org/TR/xhtml11/DTD/xhtml11.dtd">
<html xmlns="http://www.w3.org/1999/xhtml" xml:lang="fr">

XHTML transitional uses lang and xml:lang.

Front-end should take care of this for you.

The language should be a lower-case ISO2 code of the language.


<html lang="en">
<html lang="fr">
<html lang="da">
<html lang="el">

It can include a script subtag to define the script that's being used.


<!-- Japanese written in Katakana -->
<html lang="ja-Kana">
<!-- Russian written in Cyrillic -->
<html lang="ru-Cyrl">

It can include a region subtag. Usually that's the upper-case ISO2 country code.


<html lang="en-GB">
<html lang="fr-CA">
<html lang="da-DK">
<html lang="el-GR">

... but not always


<!-- Central American Spanish -->
<html lang="es-013">

All the parts can be combined but they're always in the order language-script-region.


<!-- Russian written in Cyrillic spoken in Belarus -->
<html lang="ru-Cyrl-BY">

The language alone is usually enough.

Just in-case you need to validate the language.

User Accounts

üë• üõí

Processes for interacting with a logged-in state.

For legal or financial commitments, at least one of the following is true:

Reversible
Submissions can be reversed.
Checked
Data is checked and the user can correct it.
Confirmed
The user can review, confirm and correct data before finalising.

WCAG 3.3.4 (AA) if you want to read the full text.

By having client-side and server-side validation, we automatically get Checked.

The "checkout confirmation" page means we get Confirmed.

If we do this with all user-entered data, we hit Level AAA.

WCAG 3.3.6 (AAA).

For steps in a checkout process, for example, use aria-current="step" instead of a class.


<ol aria-label="Checkout progress">
    <li>
        <a href="#billing">Billing</a>
    </li>
    <li>
        <a href="#shipping">Shipping</a>
    </li>
    <li>
        <span aria-current="step">Payment</span>
    </li>
    <li>
        <span>Confirmation</span>
    </li>
</ol>

Probably more of a front-end technique but screen readers can't read classes.

If a user is logged out automatically, they should be able to continue with no loss of data when they log back in.

For example, if a user hits "submit" and gets re-directed to the log-in screen, logging in should submit the form, or at least show the form with their data still there.

This is actually a Level AAA step, but I think Symfony does it anyway.

Time limits

‚è≥

Basically, give users enough time to use the site.

Going to give cliff-notes for this since it doesn't fully apply.

With content that changes after a set time, one of the following should be true.

  1. User can turn off the time limit.
  2. User can adjust the time limit to up to 10 times the default.
  3. User is warned at least 20 seconds before the change and can extend the time.

...

Extending should be simple, quick and obvious.

...

  1. Time limit is part of a real-time event and there's no alternative.
  2. Time limit is essential and extending it would invalidate the activity.
  3. Time limit is more than 20 hours.

Real-time event example: auction.

Invalidate example: ticket selling website.
